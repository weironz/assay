# 工单管理系统 · 技术选型文档

> 本文档整理自技术选型讨论，作为在 Ubuntu 服务器上开展开发的起点。

## 一、最终技术栈

### 前端（Web 端为主）

| 环节 | 选型 | 作用 |
|------|------|------|
| 框架 + 构建 | React 19 + Vite + TypeScript | 骨架：把数据变成界面、组件化、开发热更新 |
| UI 外观 | shadcn/ui + Tailwind CSS v4 | 皮肤：按钮/表单/表格样式与排版（源码在自己项目里，可控） |
| 数据请求 | TanStack Query | 跟后端要数据、缓存、分页、loading/错误处理 |
| 表格 | TanStack Table | 工单列表：筛选、排序、分页、虚拟滚动 |
| 全局状态 | Zustand | 少量跨页面共享状态 |

### 后端（前后端分离，独立部署）

后端核心三件套（缺一不可）：

| 组件 | 角色 | 作用 |
|------|------|------|
| NestJS | 框架（身体） | 接收请求、组织代码、跑业务逻辑、返回响应 |
| Prisma | ORM（翻译官） | 把业务操作翻译成 SQL，读写数据库；类型安全 |
| PostgreSQL | 数据库（仓库） | 真正存工单数据；状态流转/审计/复杂查询/JSON 都强 |

可选配件（按业务需要再加，第一版可不上）：

| 组件 | 用途 |
|------|------|
| Redis | 缓存、通知、SLA 超时提醒的底座 |
| casl | RBAC 权限（谁能看/改/派单，服务端强制校验） |
| BullMQ | 异步任务：SLA 超时升级、异步派单（基于 Redis） |

### 部署

- Docker Compose + Nginx，一套 compose 起全家。

## 二、请求流转（后端三件套怎么协作）

```
请求进来
   ↓
NestJS        ← 接待员，决定这个请求该干嘛
   ↓
业务逻辑        ← 例："查张三的所有工单"
   ↓
Prisma        ← 翻译官，把这句话变成 SQL
   ↓
PostgreSQL    ← 仓库，真正存数据的地方
```

## 三、最小可跑版（建议起步用这个，别被完整清单吓到）

```
前端  React + Vite + shadcn/ui
后端  NestJS + Prisma + PostgreSQL
```

Redis / casl / BullMQ / TanStack 等，等真需要了再加。工单系统第一版完全可以没有它们。

## 四、工单系统必须提前想清楚的点

1. **状态机 / 工作流**：工单核心是状态流转（待处理→处理中→待验收→关闭…）。用有限状态机 + 流转规则表实现，不要用一堆 if-else 写死。
2. **审计日志**：每次状态变更、指派、评论都留痕。设计 `ticket_history` 表。
3. **SLA / 超时提醒**：Redis + 定时任务（BullMQ）做超时升级。
4. **通知**：站内信 + 邮件 + 企业微信/钉钉 webhook，抽象成通知中心。
5. **附件**：走对象存储（MinIO 自建 / 云 OSS），别塞数据库。
6. **权限**：RBAC 多角色（提单人/处理人/主管/管理员），服务端强制校验，前端权限只是隐藏按钮挡不住人。

## 五、三端演进路线（Web → 桌面 → App）

后端只写一次，三端共用同一套 API。

| 目标 | 用什么 | 复用程度 |
|------|--------|----------|
| Web（现在） | React + Vite | —— |
| 桌面版 | Tauri（推荐，Rust 壳，体积小）或 Electron | 前端代码几乎 100% 复用，套壳即可 |
| 手机 App | React Native / Expo | 逻辑/状态/API 层全复用，UI 组件需重写 |

### 现在就要养成的「不后悔」习惯

> **把「业务逻辑/数据请求/状态」和「界面组件」分开写。**
> 获取工单列表、判断权限、状态流转等逻辑，放到独立的 hooks / service 层，别和 JSX 混在一起。
> 这样以后做 App，逻辑层直接搬走，只重写界面。这是唯一需要现在就注意的架构习惯。

## 六、关于后端用 Rust（Axum）的取舍

- 性能/类型安全顶级，但工单系统是**业务逻辑密集**（状态流转、审批、权限），不是性能密集。
- 最大短板：Rust 生态**没有成熟的工作流/BPM 引擎**（Java 有 Flowable/Camunda），审批流得自己实现状态机；所有权机制拖慢 CRUD 迭代；招人难。
- 建议：先用 NestJS 快速把业务跑通、逻辑理清；等系统稳定后，可把某个高并发模块（通知/派单）用 Rust 重写，那才是它的主场。

## 七、一句话总结

全 TypeScript 一把梭：前端 React 系吃满生态与现代工具链，后端 NestJS + Prisma + PostgreSQL 快速迭代业务，把「先进」花在刀刃上（现代工具链 + 优秀组件），而不是赌一个生态薄的框架。对三端演进也最友好。
