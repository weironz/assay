# 工单管理系统 · 开发计划

> 配套文档：[技术选型](../工单系统技术选型.md)、[设计文档](./01-设计文档.md)、[技术调研结论](./03-技术调研结论.md)

## 已确认的关键决策

- **认证**：better-auth + 服务端 Session（Redis），本地账号，预留 SSO（不用 JWT 做会话）
- **对象存储**：RustFS（单节点尝鲜）+ `StorageService` 抽象层兜底 + 附件独立备份
- **富文本**：Tiptap（WYSIWYG + Markdown 快捷输入）存 HTML，sanitize-html + DOMPurify 双端消毒
- **工单流程**：NEW/ASSIGNED/IN_PROGRESS/PENDING/RESOLVED/CLOSED（+REOPENED/CANCELLED）
- **补强设计**：队列(Queue)、工单容器↔消息分离、public/internal 可见性、requester≠assignee、分类树、标签、保存视图、SLA 倒计时、三栏详情页、状态按钮化

## 里程碑总览

| 阶段 | 目标 | 产出 | 预估 |
|------|------|------|------|
| P0 | 骨架 + docker compose 一键起 | 空跑通的前后端 + 数据库 + RustFS | 0.5~1 天 |
| P1 | 认证(better-auth) + RBAC | 登录、用户/角色/队列、Guard | 1~2 天 |
| P2 | 工单核心 + 状态机 + 消息时间线 | 建单/列表/详情/流转/回复 | 3~4 天 |
| P3 | 富文本 + 附件 + 审计历史 | Tiptap、RustFS 上传、时间线留痕 | 2~3 天 |
| P4 | 通知 + SLA（BullMQ） | 站内信、超时提醒、倒计时 | 1~2 天 |
| P5 | 前端完善 + 仪表盘 + 保存视图 | 可用的完整界面 | 2~3 天 |
| P6 | 打包 + 部署 | 生产 compose / 镜像 | 0.5~1 天 |

> 预估为单人节奏参考，非承诺。每阶段结束应能独立演示。

## P0 · 项目骨架与开发环境（先做，务必先跑通）

**目标**：`docker compose up -d` 一条命令拉起全部依赖，前后端能互相通。

- [ ] 初始化 pnpm monorepo：`pnpm-workspace.yaml`、根 `package.json`
- [ ] `apps/api`：`nest new`，接入 Prisma，连 PostgreSQL，健康检查 `GET /health`
- [ ] `apps/web`：`Vite + React + TS`，接入 Tailwind v4 + shadcn/ui，配 axios（withCredentials）+ TanStack Query
- [ ] `prisma/schema.prisma`：建全部表（见设计文档第 3 节），首次 `migrate dev`
- [ ] **StorageService 抽象**：定义接口 + `LocalFsStorage` + `S3Storage(RustFS)` 两实现，`.env` 切换
- [ ] `docker-compose.dev.yml`：postgres + redis + rustfs + adminer + api(dev) + web(dev)，源码挂载卷、热更新
- [ ] `.env.example` + 根 `.env`，跑一次 seed（4 角色 + admin + 默认队列 + 示例类型/分类）
- [ ] README：本地启动步骤

**验收**：clone 后 `docker compose up -d` → 打开 web 看到登录页；api `/health` 返回 ok；adminer 能看到表；RustFS 控制台可访问。

## P1 · 认证（better-auth）与权限（RBAC）

**目标**：能登录，会话可即时撤销，不同角色看到不同能力。

- [ ] 后端：接入 **better-auth**，会话存 Redis，httpOnly cookie；`/auth/me`
- [ ] 后端：`users` / `roles` / `queues` 模块 CRUD（仅 admin/supervisor）
- [ ] 后端：CASL 定义 ability + `PoliciesGuard`，接口按权限点拦截
- [ ] 前端：登录页、会话态（cookie，axios withCredentials）、401 拦截跳登录
- [ ] 前端：路由守卫、按角色显示/隐藏菜单与按钮
- [ ] 前端：用户/队列管理页（admin）

**验收**：admin 登录能管用户/队列；普通角色越权接口返回 403；退出登录后旧会话立即失效（验证撤销）。

## P2 · 工单核心、状态机、消息时间线（系统心脏）

**目标**：工单能建、能查、能按规则流转、能对话。

- [ ] 后端：`tickets` 模块 —— 建单（生成 `ticket_no`，requester/assignee/queue/category/priority）、列表（筛选/分页/排序/数据级权限过滤）、详情、编辑
- [ ] 后端：`WorkflowService.transition()` —— 内置流转规则表，非法流转拒绝(409)，写审计 + first_response 记录
- [ ] 后端：指派接口 `/tickets/:id/assign`（队列/个人）
- [ ] 后端：`messages` 模块 —— 回复/内部备注（type、is_internal、content_type），进时间线
- [ ] 后端：`ticket_history` 审计写入 + `/tickets/:id/history`
- [ ] 后端：分类树 `/categories`、标签 `/tags`
- [ ] 前端：工单列表（TanStack Table：筛选栏 + 分页 + 状态徽章 + 队列/优先级列）
- [ ] 前端：新建工单表单
- [ ] 前端：**三栏详情页** —— 中栏消息时间线（public/internal 底色区分）、右栏元数据、顶部状态流转按钮组
- [ ] 前端：消息回复区（先用纯文本占位，P3 换 Tiptap）

**验收**：走完 新建→指派→处理→待验收→关闭 全流程；非法流转被拒；列表按角色/队列隔离；内部备注对提单人不可见。

## P3 · 富文本、附件、审计完善

**目标**：正文富文本化、附件可传、全程留痕。

- [ ] 前端：接入 **Tiptap** 编辑器（Markdown 快捷输入、工具栏、代码块），替换 P2 的纯文本回复区
- [ ] 前端：渲染消息用 **DOMPurify** 消毒
- [ ] 后端：正文 `sanitize-html` 白名单消毒后入库
- [ ] 后端：`attachments` 模块，走 **StorageService（RustFS）** 直传/预签名 URL，存 `object_key`
- [ ] 前端：附件上传/下载、正文内联图片（走上传接口）
- [ ] 前端：操作历史时间线（Timeline）完整展示

**验收**：能富文本回复、传附件并下载、内联图片正常；粘贴 `<script>` 被消毒过滤；历史时间线完整。

## P4 · 通知与 SLA（BullMQ + Redis）

**目标**：该提醒的自动提醒。

- [ ] 后端：接入 BullMQ + Redis
- [ ] 后端：状态流转/指派/新消息时投递通知 job，写 `notifications`
- [ ] 后端：进入 IN_PROGRESS 时按类型 SLA 计算 `sla_due_at`，投延时 job；到期未解决生成超时通知
- [ ] 前端：顶部通知铃铛（未读数）、通知列表、标记已读
- [ ] 前端：**SLA 倒计时徽章**（列表/详情，临近/超时红色高亮）

**验收**：指派后被指派人收到站内信；SLA 到期产生超时通知；倒计时显示正确。

## P5 · 前端完善、仪表盘、保存视图

**目标**：界面完整、可用、好看。

- [ ] 前端：**保存筛选视图（Overviews）** —— 我的待办/未分派/已升级等预设 + 自定义保存
- [ ] 后端：`saved_views` CRUD + `/stats/overview` 聚合查询
- [ ] 前端：仪表盘（各状态工单数、我的待办、超时工单、趋势）
- [ ] 全局：加载态、空态、错误提示、表单校验统一
- [ ] 响应式适配（为将来桌面/App 做准备）+ 深色模式

**验收**：保存视图可用；仪表盘数据正确；主要页面体验顺畅。

## P6 · 打包与部署

**目标**：能部署到服务器。

- [ ] `apps/api` / `apps/web` 生产 Dockerfile（多阶段构建）
- [ ] `docker-compose.yaml`：web 走 Nginx 静态 + 反代 api，rustfs 持久化卷
- [ ] 生产环境变量、数据库迁移策略、RustFS 桶初始化、**附件备份脚本**
- [ ] 基础日志与健康检查

**验收**：目标服务器 `docker compose up -d` 一键部署可访问。

## 建议推进方式

1. **严格先做 P0**：环境不通后面全是坑。docker compose 跑起来再写业务。
2. 每阶段做完就**手动走一遍主流程**验收，再进下一阶段。
3. 状态机 + 消息时间线（P2）是核心，多花点时间把规则表、审计、可见性打牢。
4. 权限（P1）尽早接入，避免后期给每个接口补鉴权。
5. RustFS 未 GA：P3 接附件时就把 `StorageService` 抽象做扎实，本地 FS 实现同时保留，随时可回退。
