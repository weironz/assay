# 技术调研结论（2026-07）

> 本文档记录选型阶段的三项专题调研结论，供后续决策追溯。调研于 2026-07-09。

## 一、认证方案：JWT vs OAuth vs Session

### 核心澄清

**JWT 和 OAuth 不是二选一。** JWT 是「令牌格式」，OAuth 2.x / OIDC 是「授权 / 登录框架」——OAuth 流程里发的令牌*可以*就是 JWT。真正的路线之争是 **Session（有状态）vs JWT（自包含）** 做会话。

| 概念 | 层面 | 本质 |
|------|------|------|
| Session | 会话维护（有状态） | 服务端存会话，发不透明 session id（cookie） |
| JWT | 令牌格式 | 自带签名、自包含用户信息的字符串 |
| OAuth 2.x | 授权框架 | 如何在不暴露密码下授予受限访问 |
| OIDC | 认证层（OAuth 之上） | 加 ID Token（本身是 JWT）解决「你是谁」 |
| Passkey/WebAuthn | 登录因子 | 抗钓鱼的无密码方式，替代「密码」 |

### 结论：别用 JWT 做浏览器会话

2025-2026 社区共识：JWT 做 session 的硬伤是**无法即时撤销**（退出/踢人/离职失效困难）、「无状态」是伪命题（要撤销终究得存储）、存储即风险（XSS/CSRF）。JWT 的主场是**短时 API access token、服务间调用、SSO 令牌传递**。

### 决策

**better-auth + 服务端 Session（Redis）**。本地账号起步，httpOnly cookie，前端不碰 token。身份来源与会话解耦，预留未来 OIDC 对接企业 SSO/LDAP/企业微信/钉钉。better-auth 开箱带 RBAC / 2FA / Passkey，是 2026 年 React+NestJS 栈最贴合的现代认证库。

## 二、对象存储：MinIO / RustFS / 备选

### MinIO 现状（实测核实 2026-07-09）

用户判断正确，**MinIO 社区版基本已「凉」**：
- `github.com/minio/minio` 仓库**已归档（archived、只读）**。
- 2025-05/06 社区版 Web 管理 UI 被大幅阉割（账户/策略/桶/生命周期/LDAP-OIDC 登录等挪到收费 AIStor）。
- 2025-12 官方宣布社区版进入**维护模式**：不再接受新功能，安全补丁仅「个案评估」，下架预编译二进制。
- 企业版 AIStor 起步约 $96,000/年。

**结论：新项目不该再上 MinIO。**

### RustFS 现状（实测核实）

- `github.com/rustfs/rustfs`：Rust、**Apache-2.0**（比 MinIO 的 AGPL 友好）、~3 万 star、开发极活跃、定位 MinIO drop-in 替代、自带控制台。
- **但至今无 GA**（最新 `1.0.0-beta.8`），分布式/生命周期/KMS 仍标「测试中」，changelog 仍在持续修安全漏洞，开源刚满一年、无长期生产背书。

### 决策

**RustFS 单节点尝鲜 + 存储抽象层兜底。** 风险控制四条：① 单节点；② 附件独立定期备份（可重建）；③ 跟进版本升级；④ 所有读写走 `StorageService` 抽象接口（`put/get/presign/delete`），保留 `LocalFsStorage` 实现，若 RustFS 出问题可无痛切到本地 FS 或 **Garage**（成熟自托管 S3 备选）。

> 工单附件是「可重建、非强一致关键」数据，单节点试点合理。若追求绝对稳妥，`STORAGE_DRIVER=local` 一键回退本地文件系统。

## 三、参考系统调研：Zammad / GLPI / Jira Service Management

### 各系统技术栈

| 系统 | 后端 | 前端 | 富文本存储 |
|------|------|------|-----------|
| Zammad（开源 AGPL） | Ruby on Rails + GraphQL | Vue 3 + TS + Vite | HTML |
| GLPI（开源 GPL） | PHP 8 + MySQL | Twig + Vue（渐进现代化） | HTML(TinyMCE) |
| Jira Service Mgmt（商业） | Java(Spring) | React(Atlaskit) | ADF(结构化 JSON) |

### 我们已采纳的补强设计（「必须有」）

1. **工单容器 ↔ 逐条消息分离**（Zammad Article / GLPI Followup / JSM comment 共识）
2. **消息 public/internal 可见性**（内部备注对提单人不可见，底色区分）
3. **requester ≠ assignee**（显式区分提单人 / 处理人）
4. **Queue 队列层**（先归队列再派个人，派单/SLA/权限挂载点）
5. **分类树 + 标签**（统计/派单/SLA 匹配）
6. **状态流转按钮化 + 后端状态机校验**（JSM 式）
7. **SLA 倒计时可视化**（列表/详情红绿徽章）
8. **保存筛选视图 Overviews**（我的待办/未分派/已升级，agent 主工作台）
9. **三栏详情页布局**（左导航 + 中会话时间线 + 右元数据）

### 暂缓的「锦上添花」（后续迭代）

抄送人(Observer)、Urgency×Impact→Priority 矩阵(GLPI)、宏(Macros)、Task/Comment 分离、工单合并/关联、自动化规则引擎、自定义字段(先用 JSONB 兜底)、知识库、自助门户、审批流、工时(worklog)、pending 唤醒时间。

### 数据模型曾遗漏、现已补上的概念

requester≠assignee 两个人 / queue / category 树 / message.is_internal / message.type / tags / saved_views / SLA 与工作日历 / 预留 JSONB 自定义字段。

### Markdown / 富文本实现（决策）

客服/ITSM 类系统多用富文本存 HTML。我们采用 **Tiptap**（WYSIWYG + Markdown 快捷输入，Notion 式体验），提交存 HTML（`content_type='text/html'`），**双端 XSS 消毒**：服务端 `sanitize-html` 白名单入库 + 客户端 `DOMPurify` 渲染。消息表保留 `content_type` 列以便未来兼容 markdown/plain，不锁死格式。内联图片走独立上传接口，不塞 base64。
